1. Tests with NOIOMMU in i5-2310 system
=======================================

These tests were performed in a i5-2310 system which doesn't have an IOMMU using:
a. AlmaLinux 8.9
b. Kernel 4.18.0-513.9.1.el8_9.x86_64
c. Use cmem_gdb_access as no IOMMU
d. Secure boot off. The PC BIOS doesn't have the option to enable secure boot,
   and need to load the cmem_gdb_access module which isn't signed.
e. Reserve 2 GiB of physical memory for use by the cmem_gdb_access module, by using kexec to temporary reboot with
   a memmap argument:
 ~/cmem_gdb_access/reboot_with_memmap_changes.sh 2050M\$4G,1M\$6146M
About to:
sudo kexec -l /boot/vmlinuz-4.18.0-513.9.1.el8_9.x86_64 --initrd=/boot/initramfs-4.18.0-513.9.1.el8_9.x86_64.img --append="BOOT_IMAGE=(hd0,gpt2)/vmlinuz-4.18.0-513.9.1.el8_9.x86_64 root=/dev/mapper/almalinux-root ro crashkernel=auto resume=/dev/mapper/almalinux-swap rd.lvm.lv=almalinux/root rd.lvm.lv=almalinux/swap rhgb quiet memmap=2050M$4G,1M$6146M"
sudo shutdown -r now

2.2. Results from TEF1001_dma_ddr3 and NiteFury_dma_ddr3 designs with maximum number of buffers
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

The TEF1001_dma_ddr3 and NiteFury_dma_ddr3 designs were re-tested but with the max_buffer_size changed
to set the maximum number of buffers of 1023 (max DMA descriptors). This means the C2H transfers can be started
sooner, as less of wait for the corresponding H2C transfer to first write data to the DDR3 memory.

This caused a slight increase in the Mean transfer speed for TEF1001_dma_ddr3:
  - from 871.267213 (Mbytes/sec) to 873.554303 (Mbytes/sec)

Whereas for NiteFury_dma_ddr3 no measurable change (within the variation from run-to-run):
  - from 697.760850 (Mbytes/sec) to 697.583653 (Mbytes/sec)

3. Tests with NOIOMMU in W-2123 system
======================================

These tests were using the same system as the previous test, but without the IOMMU to investigate any performance
difference of having the IOMMU enabled.

To test with NOIOMMU:
a. Disabled secure boot in the BIOS which is required to Load the cmem_gdb_access Kernel module which hasn't been signed.
b. Run set_exe_vfio_capabilities.sh to set the required permissions on the executables to open VFIO devices as a
   normal user in NOIOMMU mode.
c. When booting use GRUB to manually edit the command line to:
   - Remove:
        intel_iommu=on
   - Add the following, which reserves just over half the physical memory for the cmem_gdb_access module:
        memmap=16386M\$4G,1M\$20482M

The complete command line was:
$ cat /proc/cmdline

5.2. Tests with IOMMU disabled
~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~

These tests were using the same system as the previous test, but without the IOMMU to investigate any performance
difference of having the IOMMU enabled.

To test with NOIOMMU:
a. Disabled secure boot in the BIOS which is required to Load the cmem_gdb_access Kernel module which hasn't been signed.
b. Run set_exe_vfio_capabilities.sh to set the required permissions on the executables to open VFIO devices as a
   normal user in NOIOMMU mode.
c. When booting use GRUB to manually edit the command line to:
   - Remove:
        intel_iommu=on
   - Add the following, which reserves just over half the physical memory for the cmem_gdb_access module:
        memmap=16386M\$4G,1M\$20482M
